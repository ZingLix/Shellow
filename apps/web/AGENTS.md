# Prototype Instructions

Run the local server yourself and open the preview in the browser available to this environment. Do not give the user server-start instructions when you can run it.

Before making substantial visual changes, use the Product Design plugin's `get-context` skill when the visual source is unclear or no longer matches the current goal. When the user gives durable prototype-specific design feedback, preferences, or decisions, record them in `AGENTS.md`.

When implementing from a selected generated mock, treat that image as the source of truth for layout, component anatomy, density, spacing, color, typography, visible content, and hierarchy.

## Shellow landing-page decisions

- Use the approved Sillage landing-page reference for structure: chapter-based sections, restrained motion, and real device captures.
- Preserve Shellow's existing palette from the native apps: `#0D0F0E`, `#151817`, `#1C9F70`, `#45D18C`, `#E0E8DE`, `#95A39B`, and `#EDAD38`.
- Position Shellow first as a reliable, high-performance native mobile SSH terminal. Only describe coding agents that the shipped clients currently implement; Codex is supported today and other agents must be labeled as planned until implemented.
- The page narrative is reliability, native performance, coding agents, dual-platform support, and open source.
- Do not display real hostnames, usernames, project paths, credentials, or personal task content. Marketing screenshots must use isolated simulator data.
- Keep Magic UI effects purposeful and sparse. Avoid generic SaaS gradients, oversized rounded cards, fake metrics, and unsupported performance claims.
