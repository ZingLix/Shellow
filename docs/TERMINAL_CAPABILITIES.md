# Shellow Terminal Capability Checklist

This is the working checklist for a terminal that is good enough to use daily, not just a command demo.

## Input

- Direct byte input into the terminal stream.
- Enter, Backspace, Tab, Escape, arrow keys, Home/End, and PageUp/PageDown.
- Local command history recall with Up/Down while editing a normal shell prompt.
- Function keys F1-F12 using common xterm SS3/CSI sequences for TUI applications.
- Application cursor-key mode (`DECCKM`) so full-screen apps can switch arrow keys from CSI (`ESC [ A`) to SS3 (`ESC O A`) sequences.
- Common control combinations: Ctrl-C, Ctrl-D, Ctrl-L, Ctrl-Z, Ctrl-[, plus readline-style Ctrl-A/Ctrl-E/Ctrl-K/Ctrl-U/Ctrl-W/Ctrl-R.
- Alt/Meta input using ESC-prefixed key sequences for readline and TUI shortcuts.
- Pasted text sent exactly to the terminal stream.
- Confirmation for risky pastes such as multiline or large clipboard payloads.
- Bracketed paste mode (`DECSET 2004`) so shells and full-screen apps can distinguish pasted text from typed input.
- Hardware keyboard support through native key events.

## Interaction

- Long-lived PTY shell sessions rather than one command per connection.
- Optional per-host tmux, GNU screen, or Zellij persistence that creates or attaches to a named session after SSH connects.
- Full-screen programs such as `vim`, `nano`, `less`, `top`, and `tmux`.
- Terminal resize events mapped to PTY window-change messages.
- App-level clear visible terminal and reset local terminal state controls.
- Cursor position, alternate screen, scrollback, and selection semantics.
- Auto-follow latest output and a one-tap jump back to the bottom after reviewing scrollback.
- Dynamic terminal titles through OSC 0/2 sequences emitted by shells and TUI programs.
- Terminal bell (`BEL`) events surfaced in the mobile UI.
- Mouse reporting for touch-driven TUI interaction, including xterm normal mouse mode, button-motion mode, release events, and SGR coordinates.

## Text Operations

- Copy selected text, including row/range selection and drag-based character-range selection.
- Copy current visible terminal contents.
- Save/export the current visible terminal transcript as a plain UTF-8 text file.
- Paste from the platform clipboard.
- Remote-to-local clipboard requests through OSC 52, gated by user confirmation.
- Search visible terminal contents with highlighted matches and previous/next navigation.
- Detect and copy URLs from selected visible terminal output.
- Confirm multiline or large paste payloads before sending them to the terminal.
- Wrap paste payloads with bracketed paste markers when the terminal application enables bracketed paste mode.
- Preserve monospace layout and wide-character cells.

## Rendering

- VT escape parsing through the official `libghostty-vt` terminal core.
- GPU-backed grid rendering with a path toward glyph-atlas text and dirty-row uploads.
- Dirty-row redraws, cursor shape rendering, and selection highlights.
- Persisted display controls for terminal font size, line height, and app color-scheme preference.

## SSH Basics

- Password and private-key authentication.
- Platform-backed secret storage for passwords, private keys, and key passphrases.
- Host-key verification and fingerprint pinning.
- Disconnect/reconnect lifecycle.
- Keepalive and network error surfacing.

## Current Implementation Status

- Direct input stream: implemented in the shared Rust core and wired to iOS/Android. The mobile terminal viewports now own hidden native input responders, so tapping the terminal area sends soft-keyboard text, Enter/newlines, Backspace, paste payloads, and hardware-key events directly into the terminal stream instead of through a visible staging input field.
- Local command history recall and reverse search: implemented in the shared Rust core for normal prompt input. Up/Down browse previously committed local commands, preserve the in-progress draft while browsing, and let recalled commands execute directly with Enter. `Ctrl-R` enters a readline-style `(reverse-i-search)` prompt, filters history as the user types, repeated `Ctrl-R` cycles to older matches, Enter executes the current match, and Esc restores the draft. Recalled/search prompt text is rendered by the terminal snapshot itself; iOS and Android no longer depend on a visible input field mirror for normal typing.
- Special keys, navigation keys, and common Ctrl combinations: implemented in iOS and Android toolbars. The shared Rust prompt owns readline-style local cursor editing for Left/Right/Home/End, Backspace/Delete, and `Ctrl-A`/`Ctrl-E`/`Ctrl-K`/`Ctrl-U`/`Ctrl-W`; `Ctrl-R` remains the shared reverse-search path. iOS direct input capture also handles hardware arrows, Home/End, PageUp/PageDown, and common Ctrl combinations.
- Alt/Meta input: implemented as a one-shot `Alt` toolbar modifier on iOS and Android, sending ESC-prefixed text or special-key sequences. iOS hardware Option+text input is also mapped to ESC-prefixed Meta text.
- Function keys F1-F12: implemented in iOS and Android terminal toolbars. The shared Rust demo decodes SS3 F1-F4 (`ESC O P` through `ESC O S`) and xterm-style CSI F5-F12 (`ESC [ 15~` through `ESC [ 24~`) so full-screen apps can receive the same sequences.
- Application cursor keys: implemented in the shared Rust VT grid snapshot via `libghostty-vt` application-cursor mode state. iOS and Android toolbar arrows switch between CSI and SS3 sequences while the mode is active; iOS hardware arrows follow the same mode.
- Copy current screen and paste clipboard: implemented on iOS and Android.
- Risky paste confirmation: implemented on iOS for multiline/large paste payloads, controlled by the existing Confirm Paste setting. Implemented on Android with a safety-default confirmation for multiline/large paste payloads.
- Bracketed paste: implemented in the shared Rust VT grid snapshot via `libghostty-vt` bracketed-paste mode state. iOS and Android wrap paste payloads with `ESC[200~` and `ESC[201~` while that mode is active.
- Dynamic title updates: implemented in the shared Rust core through `libghostty-vt` OSC 0/2 parsing. Local demo, remote exec output, and live SSH output can update the session title shown by iOS and Android.
- OSC 52 remote clipboard requests: implemented in the shared Rust core by parsing OSC 52 payloads with BEL or ST terminators, decoding the clipboard payload, ignoring query requests, and exposing `pending_clipboard_text` plus a monotonic `clipboard_sequence` in the shared snapshot. iOS and Android show a `Remote Clipboard` confirmation before writing the decoded remote text to the platform clipboard.
- Terminal bell: implemented in the shared Rust core through `libghostty-vt` BEL callbacks. Local demo, remote exec output, and live SSH output increment `bell_count`; iOS shows a compact, accessible bell-count badge in the terminal header, and Android shows a `Bell N` header indicator.
- Mouse reporting: implemented in the shared Rust grid snapshot by tracking xterm mouse private modes (`DECSET 1000/1002/1003`), a dedicated `mouse_drag_reporting` flag for button-motion modes (`1002/1003`), and SGR mouse mode (`DECSET 1006`). iOS and Android turn row taps into SGR mouse press sequences, and turn row drags into SGR press/drag/release sequences while button-motion reporting is active. The local `shellow mouse` demo enters alternate screen, enables normal mouse + button-motion + SGR mouse reporting, and echoes received press, drag, and release coordinates.
- Host-key fingerprint pinning: implemented in the shared Rust SSH layer. Host profiles can carry an optional SHA256 host-key fingerprint; iOS and Android pass it through FFI/JNI; `russh` rejects the connection when the server public-key SHA256 fingerprint does not match. Profiles without a pin remain usable but are surfaced as `Host key unverified` / `host-key=unverified (no SHA256 pin)` instead of silently pretending to be verified.
- Automatic TOFU capture: implemented for live SSH handshakes. The `russh` client handler records the server public-key SHA256 fingerprint even when no expected pin is configured, successful live PTY sessions expose it as `observed_host_key_sha256` in the shared snapshot, and iOS/Android automatically persist it back to the matching unpinned host profile and reconnect target.
- Keepalive policy: implemented for live PTY SSH sessions through `russh` client config. Long-lived shell sessions set `keepalive_interval=30s`, `keepalive_max=3`, and TCP `nodelay=true`; terminal preview rows surface `keepalive=30s max-missed=3` so the active policy is visible.
- Live PTY shell: implemented for password and OpenSSH private-key authentication via `russh`. Private-key profiles open a mobile entry sheet for the OpenSSH private key, optional passphrase, and startup command; the shared Rust core parses unencrypted keys and decrypts encrypted keys with the provided passphrase before `authenticate_publickey`.
- Rich SSH error surfacing: implemented in the shared Rust SSH layer. Private-key auth now preflights OpenSSH key parsing before starting a live shell, reports encrypted-key/passphrase/decrypt failures distinctly, reports connection/authentication/signature-query phases separately, and surfaces host-key mismatch as `expected` versus `got` SHA256 fingerprints when russh observes the server key.
- VT grid snapshots: implemented in Rust with `libghostty-vt`, including cursor position, active primary/alternate screen, visible rows, styled cells, terminal modes, and primary-screen scrollback rows.
- Dirty-row redraw bookkeeping: implemented in the shared Rust grid snapshot as `dirty_rows`. The first grid frame marks all visible rows dirty; subsequent snapshots compare text, styled runs, cursor position/shape/visibility, screen kind, scrollback length, and dimensions so only changed rows are exposed to iOS/Android. Mobile poll loops ignore dirty-only clean frames so the UI keeps the last real redraw hint instead of being overwritten by passive polling.
- Cursor shapes: implemented in the shared Rust grid snapshot through `libghostty-vt` render state. iOS and Android render block, underline, and bar cursor markers from the shared `cursor_shape` field.
- Styled cell runs: implemented in Rust and rendered on iOS/Android for ANSI colors, 256-color, truecolor, background color, bold, underline, strikethrough, and inverse-video cells.
- Alternate-screen rendering: implemented on iOS and Android. The local `vim` demo enters an alternate screen, accepts text/special keys, renders a block cursor, copies visible grid text, and exits with Esc. The local `less`/pager demo also enters alternate screen, keeps the app viewport top-anchored instead of bottom-following, supports Up/Down, PageUp/PageDown, Home/End, Space/b, and exits with `q` or Esc. Local `nano`, `top`/`htop`, and `tmux` demos now exercise additional full-screen terminal contracts: control-key exits/status updates (`^O`, `^X`), cursor-key updates, `q` exit, and tmux `^B` prefix handling.
- Row-range terminal selection: implemented on iOS and Android for both history rows and VT grid rows, with highlight, Copy Selection, and Clear Selection actions. Tapping a second row of the same surface extends the selection range from the anchor row.
- Character-range terminal selection: implemented on iOS and Android for VT grid rows. Tapping still selects whole rows for quick copy; dragging across a grid row records terminal cell columns, highlights only the selected cell range, and copies only the selected substring. The selection math uses terminal cell widths so CJK double-width characters, emoji, and combining marks are copied as intact grapheme clusters rather than byte slices.
- Search visible terminal contents: implemented on iOS and Android for both history rows and VT grid rows. The mobile UI exposes a search field, match counter, previous/next navigation, active-match scrolling, row-level fallback backgrounds, and terminal-cell-range renderer highlights without sending anything into the terminal input stream.
- Visible URL quick copy: implemented on iOS and Android. Selected terminal text is scanned for the first HTTP(S) URL; iOS uses `NSDataDetector`, Android uses an HTTP(S) URL matcher, and `Copy Link` copies only the URL to the platform clipboard. The local `shellow links` demo emits stable example URLs for verification.
- Primary-screen scrollback: implemented in the shared Rust grid snapshot by merging Ghostty scrollback rows before the visible viewport. iOS and Android render, copy, and row-select the combined grid rows; alternate-screen apps still render only the active alternate screen.
- Auto-follow and jump-to-bottom: implemented on iOS and Android. The iOS history and grid viewports follow newly appended output only while the reader is already near the bottom, preserve the reading position while reviewing scrollback, reveal the bottom-jump control only when it is needed, and return to the live edge as soon as the user starts typing. Android follows new output while search is inactive and exposes its bottom-jump control for returning to the newest output after reading scrollback.
- UTF-8 and wide-character demo: implemented in Rust through `shellow wide`, using the same VT grid snapshot path for CJK double-width text, mixed Unicode prompts, box drawing, combining marks, and emoji.
- Hardware and direct viewport text entry: implemented through native key events on iOS and Android, including ordinary text, Enter/Tab/Esc, arrows, Home/End, PageUp/PageDown, Delete/Backspace, function keys, common Ctrl combinations, and Alt/Meta ESC-prefixed text. iOS uses a hidden `UIKeyInput` responder attached to the terminal viewport; Android uses a hidden focused Compose `BasicTextField` sentinel as the IME bridge. Normal local-prompt editing is rendered from the shared terminal snapshot, including cursor movement, middle insertion/deletion, word delete, line kill, and home/end editing. The former visible staging input field is no longer part of the primary terminal path, and the mobile toolbar includes `^B`, `^O`, and `^X` so tmux/nano remain usable without an external keyboard.
- App-level clipboard/search hardware shortcuts: implemented as Cmd-C/Cmd-V/Cmd-F on iOS and Ctrl-Shift-C/Ctrl-Shift-V/Ctrl-Shift-F on Android, keeping terminal Ctrl-C/Ctrl-F available for the remote program.
- PTY resize: implemented in Rust and wired to iOS/Android viewport measurement; live SSH sends window-change messages.
- Clear and reset controls: implemented in the shared Rust core and wired to iOS/Android. Clear removes visible history/grid output while preserving the current session, terminal size, and connection state; Reset exits local alternate-screen/demo modes, clears pending input and local display state, and returns to a clean prompt without dropping the session handle.
- Transcript export: implemented on iOS and Android using the same visible terminal text model as Copy Terminal. iOS saves UTF-8 `.txt` files under the app Documents `Shellow-Transcripts` directory and shows a saved-file confirmation. Android saves UTF-8 `.txt` files under the app-specific Documents `Shellow-Transcripts` directory and shows a saved-file confirmation dialog.
- Disconnect/reconnect lifecycle: implemented through the shared Rust core and FFI/JNI bridge. iOS and Android terminal headers expose Disconnect while a session is active, calling `disconnect_live_shell` and returning the session to `disconnected`. After connecting from a saved host profile, both apps keep an in-memory reconnect target for the current process and expose Reconnect when disconnected.
- Persistent multiplexer sessions: implemented on iOS and Android as an opt-in, persisted host-profile setting with a choice of tmux, GNU screen, or Zellij and a validated shell-safe session name. Existing saved iOS tmux JSON remains compatible and decodes with tmux as the default backend. After SSH reaches the connected state and the login prompt settles, Shellow runs the selected backend's attach-or-create flow: `tmux new-session -A -s <name>`, exact-name GNU screen lookup followed by `screen -D -R <pid.name>` or `screen -S <name>`, or `zellij attach --create <name>`. Reconnect reuses the same startup command. Terminal tools expose backend-specific sessions, windows/tabs, panes/regions, switch/create, and detach controls with delayed prefix dispatch after the native menu closes. Hosts missing the selected executable fall back to the regular SSH shell with an explicit message instead of failing the connection. Real attach/create, switching, detach, reconnect, and content/layout recovery are proven end-to-end on both the iOS simulator and an Android PJD110 device against a reachable SSH host with all three multiplexers installed.
- Target capability detection: implemented on iOS and Android through a separate read-only SSH exec session so probing never enters, detaches, or writes into the user's interactive multiplexer session. Password and private-key authentication both have dedicated FFI plus Swift/JNI bridge paths. The probe detects operating-system name/version, kernel/release, architecture, login shell, and installed tmux/screen/Zellij versions; it augments non-login PATH with common Homebrew, Linuxbrew, and user-bin locations. Support grades are based on the exact Shellow prerequisites advertised by the target: tmux `new-session -A` in `list-commands`, screen `-R`, and Zellij `attach --create`. Reports persist in the host profile, refresh automatically after 24 hours when saved credentials are available, and can be refreshed manually. Grades are `Full`, `Limited`, or `Not installed`; an unavailable component remains selectable because the host may change and the regular-shell fallback is still safe.
- Persisted host profiles and host-key pins: implemented on iOS with `UserDefaults` JSON storage and on Android with `SharedPreferences` JSON storage. Added profiles preserve their optional SHA256 host-key pins across app launches.
- Platform-backed SSH secret storage: implemented on iOS with Keychain generic-password items keyed by stable host-profile UUID and secret kind (`password`, `privateKey`, `passphrase`). Implemented on Android with Android Keystore AES-GCM encryption and `SharedPreferences` ciphertext records keyed by stable host-profile id and secret kind. Password and private-key sheets on both platforms can save newly entered secrets and reconnect with a blank field when a saved secret exists, without writing secret material into host-profile JSON.
- Persisted terminal display settings: implemented on iOS with `UserDefaults` JSON storage for font size, line-height scale, paste/toolbar preferences, keepalive preference, and app color-scheme preference. iOS terminal viewport resize calculations, SwiftUI grid rows, and Rust surface sizing all use the persisted font size and line-height scale. Implemented on Android with `SharedPreferences` display settings for terminal font size and line-height scale; the Compose grid viewport, transparent hit-test rows, history rows, and PTY resize calculation all use the same settings.
- Native rendering surface MVP: implemented on iOS with an `MTKView`/`CAMetalLayer` terminal grid surface and on Android with a `SurfaceView`/`ANativeWindow` host plus transparent Compose interaction/accessibility rows. The current iOS grid path attaches the layer to Rust, creates/configures a `wgpu::Surface`, uploads the shared glyph atlas through an explicit raster backend boundary, builds terminal cell/glyph vertices from the current grid snapshot, sends selection/search cell ranges through `shellow_engine_set_renderer_overlay_json`, and asks Rust to present visible terminal frames through `shellow_engine_render_frame_viewport_json`. Android now converts the platform `Surface` to a retained `ANativeWindow` in JNI, attaches it through the shared renderer ABI, maps scrollback to a visible `firstRow/rowCount` viewport, sends viewport-relative selection/search overlay JSON before rendering, asks Rust to present frames through the same viewport render-frame API, and keeps Compose terminal grid rows transparent so terminal visuals do not fall back to Canvas/Skia. The shared Rust core owns the persistent renderer runtime exposed through `shellow_engine_render_frame_viewport_json`, `shellow_engine_renderer_info_json`, and `shellow_engine_set_renderer_overlay_json`; it consumes the current terminal grid/history plus renderer overlay ranges, reports cell metrics, dirty rows, frame signatures, active screen, styled-run counts, native-surface overlay range counts, `glyph_atlas_backend`, `glyph_atlas_target_backend`, `glyph_atlas_real_font_ready`, `glyph_layout_backend`, `glyph_layout_target_backend`, `glyph_layout_shaping_ready`, per-frame glyph-layout cluster counts, and shaped glyph counts, initializes one native `wgpu` device/queue on first frame, prefers a real `fontdue-system-font-rasterizer` atlas from `SHELLOW_RENDERER_FONT_PATH` or platform monospace system fonts, falls back to `procedural-cell-rasterizer` only when no parseable font is available, runs `rustybuzz-terminal-shaper` in native/font builds to convert text into HarfBuzz glyph IDs and terminal-cell ranges, stores shaped `FontGlyph(u16)` atlas keys, rasterizes shaped glyph IDs with `fontdue::rasterize_indexed`, keeps `terminal-cell-cluster-layout` as the no-shaper fallback, uploads the glyph atlas texture with `write_texture`, packs dirty rows into a GPU buffer with `write_buffer`, reuses the device on later frames, accepts native surface attach descriptors through FFI/JNI, and presents native-surface or offscreen `wgpu` frame passes in native builds.
- Native renderer surface attach/presentation ABI: implemented for iOS terminal frames and wired for Android terminal frames. iOS passes the active `MTKView` `CAMetalLayer` pointer and drawable dimensions into `shellow_engine_attach_core_animation_layer_json`; Rust creates/configures the `wgpu::Surface`, presents the attach probe, then presents terminal content frames from the shared renderer when the grid host is mounted. Android passes a real `SurfaceView` `Surface` into JNI, JNI converts it with `ANativeWindow_fromSurface`, retains/releases that native window with the engine lifecycle, and forwards the pointer into `shellow_engine_attach_android_native_window_json`; Rust's Android native build creates/configures the matching Vulkan `wgpu::Surface` from `AndroidNdkWindowHandle`. Rust tracks the attachment generation, surface kind, dimensions, platform support, configuration state, probe presentation state, and terminal-frame presentation state separately.
- Remaining rendering work: verify the Android `SurfaceView`/`ANativeWindow`/Vulkan presentation path on a reachable device, tune z-order/frame pacing, and extend `rustybuzz-terminal-shaper` with fallback font chains, emoji/color glyph handling, RTL/bidi terminal policy, and tighter terminal-cell positioning.
- Remaining VT backend work: prove the vendored `libghostty-vt` Zig build on an Android device target, then evolve the adapter from per-snapshot terminal reconstruction toward persistent terminal state where it improves live-session performance. The iOS simulator path is now proven through the native `ShellowCore.xcframework` build and live runtime.
- Remaining SSH verification work: Android live SSH/TOFU runtime proof still needs a reachable device run. iOS simulator live SSH is proven against a reachable password-auth host, including unpinned first-handshake capture and profile pin persistence.
